\section{Designing robust \ac{sdg} frameworks}
\label{sec:methodology}


\subsection{Design Rules}
\label{sec:methodology:designrules}

While \acf{sdg} is a mechanism frequently used in research, very few generalized tools have been developed to facilitate researchers in generating discussions. Concordia~\cite{concordia} is one of the very few frameworks dedicated to managing interactions between LLMs. However, it has to our knowledge been used in exactly one study~\cite{navarro2024} and one competition~\cite{neurips_concordia} since its initial release in 2023---most research is built on purpose-built tools (\S\ref{sec:related:discussions}). Given that the ecosystem around \ac{sdg} is maturing rapidly after the introduction of LLMs in 2022~\cite{adornetto2025_gabms}, it is noteworthy that standardized tooling has not been adopted at any large scale by the research community. What makes these frameworks widely used by other people has not been explored in literature, despite many such implementations. By observing which systems worked and which failed, we propose the following design rules:

\begin{enumerate}
	\item \textbf{The framework must be as simple as possible, while achieving the researchers' goals}. In \ac{cs} there is a widely shared notion that simple systems are almost always better at performing their functions (``Keep It Simple Stupid''---KISS) \cite{beck2000extreme,thomas2025simplicity}, which has been validated in real-world applications \cite{banker1989software,ogheneovo2014relationship}. The violation of this rule could explain issues in other academic \ac{sdg} systems~\cite{balog_2024}. Complex and inflexible architecture, as well as initial computational requirements and running costs have also been mentioned as a limiting factor for Concordia~\cite{neurips_concordia}.
	
	\item \textbf{Complexity is directly related to researcher bias}. Each new feature necessarily follows our own expectations with how human discussions work. Indeed, recent work on LLM behaviors~\cite{park2023game} managed to derive interesting insights exactly because it did \emph{not} tamper with the way LLM users interacted. This is generally supported by \ac{gabm} literature~\cite{adornetto2025_gabms}.
	
	\item \textbf{\ac{sdg} should be cheap} The primary reason for designing and using \ac{sdg} methodologies is to bypass the monetary and time cost of human experiments~\cite{rossi_2024}. As the data obtained from these methodologies are inherently less valuable than those gathered through live human interactions, it is necessary to generate large quantities of data without compromising cost efficiency, and thus scalability.
\end{enumerate}


\subsection{Context Management}
LLM capabilities have incentivized researchers to create elaborate ways of managing context such as dynamic summarization \cite{balog_2024,park2023game}, LLM self-critique \cite{yu_2024_fincon}, or memory modules \cite{Vezhnevets2023GenerativeAM}. These approaches however are not explainable, and require substantial computational resources either due to repeated generation~\cite{balog_2024,adornetto2025_gabms}, or due to the loading of specialized modules in (very valuable) GPU VRAM memory~\cite{Vezhnevets2023GenerativeAM}, rendering them inadequate for tasks where scalability and capital intensity are limiting factors.

These approaches may be appropriate for highly specialized settings, but they are not necessary for every task. Prior work on online discussions has demonstrated that providing the $h$ most recent comments as context is sufficient for LLM participants to engage meaningfully \cite{pavlopoulos_2020_toxicity}. While this strategy may not be universally optimal, we argue that it is necessary in certain cases and adequate in many. In online discussions in particular, the aforementioned context-management techniques \emph{should not be used}, as they inevitably abstract away critical elements such as writing style, precise wording, and tone—features that are central to interpreting participant stance and posturing. More broadly, directly including the contents of prior interactions preserves all available information while maintaining scalability (assuming relatively small context windows) and ensuring transparency and explainability.


\subsection{Turn Taking} 
Turn-taking refers to the process with which humans take turns speaking in a discussion. The goal is typically to minimize idle time in discussions, while letting most participants participate. This process is poorly understood---only experimental coarse-grained solutions have been devised for automated systems~\cite{frai_turn_taking}. While humans typically learn when to speak after years of constant social interaction~\cite{frai_turn_taking}, this process must be implemented for LLM agents which are trained as chatbot assistants---and therefore to always respond. 

One solution is to use a dedicated smaller model whose only task is to decide whether a larger model should speak, but this approach has been only demonstrated in narrow application settings~\cite{chen-etal-2025-llamapie}. Concordia~\cite{Vezhnevets2023GenerativeAM} uses a centralized system where a single agent (``the Game Master'') determines who should speak, ignoring input from the participants themselves. This leads to some computational overhead, as the Game Master has to decide the next course of action using prompting. Deliberate Lab~\cite{Tsai_Deliberate_Lab_Open-Source_2025} uses a system where each agent is assigned a typing speed, which determines priority. These approaches showcase the necessity of a turn-taking system when using LLMs as participants.

In online fora users often create ``comment chains'' following-up on responses to their previous comments. For each discussion turn we either allow the previous user to respond (with a $40\%$ probability), or select another random participant ($60\%$). This probability was selected experimentally; larger values tend to create ``debate''-style discussions between only two or three participants, while lower values tend to create scenarios with minimal interaction between them.\footnote{It would be interesting to analyze quantitatively how discussions change when modifying this parameter, but this is outside the scope of our work.} Our turn-taking function is thus defined as:
\[
t(i) = \left\{
\begin{array}{ll}
	\textit{unif}(U) & i=1, i=2\\
	\textit{unif}(U \setminus \{t(i-1)\}) & i > 2, p=0.6 \\
	t(i-2) & i > 2, p=0.4 
\end{array} 
\right.
\]
\noindent where $U$ is the set of all non-facilitator users, \textit{unif} is a function sampling from the uniform distribution, and $p$ represents the probability of the corresponding option being selected. When a facilitator is present, $t$ alternates between picking a normal user and the facilitator, who can respond after every comment, or stay silent by emitting an empty string. 


\subsection{Instructions} 
Playing out scenarios is not a task typically encountered during LLM instruction-tuning---therefore we need to give explicit instructions on how they should behave. A simple prompt would be to instruct the LLM to behave as a human would; this however is insufficient by itself. Due to alignment procedures LLMs exhibit extreme agreeableness~\cite{park2023game,anthis_2025}, which leads to repetitive scenarios where agents show a narrow band of ``behaviors''. LLM finetuning could be used to alter this behavior, as was the case in political science studies using LLMs~\cite{Taubenfeld2024SystematicBI}. This however is expensive in two main ways; first, the process itself requires extensive computational researchers, and secondly, both the finetuned model and another model for the facilitator would need to be loaded in memory, leading to significant capital requirements for researchers.

We could instead use prompting by instructing participants to respond to repeatedly toxic comments. This is an example of the trade-off between complexity and research bias v.s. the need to acquire meaningful data (\S\ref{sec:methodology:designrules}). Facilitators are only instructed to respond only when necessary, besides specialized instructions following established facilitation strategies (see \S\ref{ssec:experimental:strategies}). We also assign antagonistic motivations to a subset of participants (trolls), as different ``factions'' in discussions can lead to interesting and less repetitive emergent conversational dynamics~\cite{abdelnabi_negotiations}.

 It is well known that LLMs tend to align more closely with WEIRD (Western, Educated, Industrialized, Rich, and Democratic) and socially prominent groups, while diverging from minorities~\cite{sap-etal-2022-annotators,goyal2022_toxicity,gallegos-etal-2024-bias} due to their nature as statistical models that adopt the bias of their training data during pretraining. These biases are further shaped by alignment procedures and prompt sensitivity~\cite{liu2023,santukar_2023}. Thus, synthetic discussions with out-of-the-box LLMs typically result in similar opinions and arguments, with no variance. 
 
 Including \acp{sdb} (e.g., gender, age and education) in prompts has proven promising in the generation of varied content \cite{burton2024large,sommerauer-etal-2025-simulating}. These ``persona'' prompts essentially shift the distribution of output tokens, hence producing content of larger variety when using multiple distinct personas~\cite{liu_2024_bias}. It is worth noting that these shifts in speech do not necessarily correlate to the actual human groups these prompts are modeling~\cite{lutz-etal-2025-prompt,liu_2024_bias}, but this is not concerning for our use-case since we cannot claim to simulate human behaviors with LLMs (\S\ref{ssec:related:human-llm}).

The creation of these personas is sometimes implemented by randomly selecting sociodemographic attributes from a number of dimensions such as age, gender, employment etc.~\cite{adornetto2025_gabms} We instead pick personas from the PERSONA dataset~\cite{castricato-etal-2025-persona}, which includes a curated set of personas based on US census data (see Appendix~\ref{ssec:app:sdbs} for details).\footnote{We use the public version of the dataset, since we were unable to establish contact with the authors for gaining access to the full dataset.} We add random employment information to these personas, since we experimentally discovered that doing so improves discussion variety. Using these sociodemographic prompts, we observe that LLM users are able to create and share personal narratives and experiences from the provided information (Fig.~\ref{fig::example}).

In general, a \ac{gabm} system should be validated based on its purposes and goals, as well as the availability of relevant data~\cite{collins2024}; \cite{adornetto2025_gabms} posit that systems that only exist to further research should not be held to the same standard as those used for policy-making or medicine for example.

There is currently no consensus on how a \acp{gabm} system should be evaluated~\cite{adornetto2025_gabms}. We can however use a subset of foundational methods from non-generative agent-based models~\cite{collins2024}. We can also use ``face validation''~\cite{rand2011_rigor}, which employs humans to judge observed behaviors and dynamics. Specifically, since the original procedures were designed with Machine Learning and rules-based models in mind, we use the adapted guidelines proposed by~\cite{adornetto2025_gabms} for generative agents. A list of possible evaluation foundational (1-4) and face-validation (5-7) strategies with the viability for \ac{sdg} is given below:

\begin{enumerate}
	% Foundational
	\item \textbf{Data analytics} is used extensively throughout our evaluation pipeline to extract useful statistics from our simulations.
	
	\item \textbf{Docking} compares the outputs of the evaluated \ac{gabm} to a state-of-the-art \ac{gabm}. Since we are one of the first to propose a generalized framework for \ac{sdg}, there is no apparent ``state-of-the-art'' model to be compared against.
	
	\item \textbf{Empirical validation} compares real-world data to ensure to the model's predictions. We can certainly use computational metrics such as \textit{Diversity}~\cite{ulmer2024} (\S\ref{sec:related:quality}) to gauge the quality of our synthetic discussions. Low diversity points to pathological problems (e.g., LLMs repeating previous comments). Conversely, extremely high diversity may indicate a lack of interaction between participants---a discussion where participants engage with each other will naturally feature some lexical overlap, including common terms or paraphrasing points made by others. We can thus compare the distribution of diversity scores for synthetic discussions with that measured on sampled human discussions to estimate the former approximate real-world content variety and participant interaction. In our case, there is also a general lack of data for online discussion facilitation~\cite{korre2025evaluation}; thus we can not examine the behavior of LLM facilitators besides some general trends.
	
	\item \textbf{Sampling} and \textbf{visualization} lead to prohibitive computational costs in the case of the former, and subjective biases for the latter~\cite{adornetto2025_gabms}.
	
	% Face validation
	\item \textbf{Survey-based validation} interviews the agents themselves. We do not follow this approach, because the large amount of agents (44 agents, 7 active at any time, with randomly assigned role prompts) used makes evaluation and analysis impractical from the perspective of time and complexity. It is also unclear what information we could derive from these agents.
	
	\item \textbf{Observational studies} analyze human behavior in similar settings, and compare them with the generative agents. We use this technique by observing emergent social behaviors that are well-accounted for in literature.
	
	\item \textbf{Expert reviews} are not used, since the difficulty and cost of finding experts in the field of facilitation and social interactions is time consuming and non-scalable. 
\end{enumerate}

The selection of a subset of validation methods for \textit{inferential} agents (who only make decisions) based on the system's goals and limitations is often used in literature, and usually accepted by the research community~\cite{adornetto2025_gabms}.


\section{Scalability}
\label{ssec:methodologies:cost}

One of the defining arguments for the development and use of \ac{sdg} frameworks is the ability to scale up experiments. This is achieved in two ways; speed and cost. The former is intuitive, since we no longer need to recruit, orient and instruct human participants. Unlike humans, LLMs also can completely forget previous experiments, making successive experiment runs independent between each other. The latter is also mostly achieved by removing human participants from the loop. However, we need to take into account that the data we get from synthetic discussions are of strictly worse ``quality'' and less utility than human discussions. To partially offset this inherent limitation, we should try minimizing the costs associated with these experiments.


\subsection{Calculating Cost}

An exact cost derivation is impossible due to the sheer amount of variables influencing LLM inference cost, but nevertheless, an estimation remains invaluable for experiment design. Since there is no standardized methodology for comparing the costs of human and synthetic experiments, we extend a proposed methodology which compares on-premises LLMs with cloud-based proprietary models.

The nature of the costs involved in any of the three methods are completely different:
\begin{itemize}[nosep,noitemsep]
	\item Open-source models incur cost from utilization of core infrastructure, electricity consumption,and asset (GPU) depreciation. 
	\item Proprietary models charge a set amount of cost per input and output token.
	\item Human participants cost wages and platform fees.
\end{itemize}

Any cost comparison thus necessitates normalizing costs to a universal metric: the \acf{cpt}, which estimates the monetary cost of completing an arbitrary task. In our experimental setup, a task corresponds to the completion of one full discussion, which may consist of multiple sequential model requests.

\paragraph{Open-source models}
Self-hosting open-source LLMs requires calculating the \ac{tco}, which converts fixed and recurring expenditures into a variable cost per unit of output \cite{pan2025_cost_benefit}. \ac{tco} encompasses both \ac{capex} and \ac{opex} amortized over the asset's lifespan. This covers the case of self-hosting on-premises, as well as the case of using cloud hosting services (such as Amazon Bedrock).

The general formula for calculating the Total Cost of Ownership ($\text{TCO}_{\text{Self-Hosted}}$) over $N$ annual periods is defined as the sum of annualized capital and operational costs \cite{pan2025_cost_benefit}:

\[
\text{TCO}_{\text{Self-Hosted}} = \sum_{n=1}^{N} \left[ \text{CapEx}_n + \text{OpEx}_n \right]
\]

\noindent Where $\text{CapEx}_n$ includes the initial hardware cost (e.g., GPU servers) amortized over a depreciation period (often 4 years),and $\text{OpEx}_n$ includes yearly hosting, power,  cooling, and personnel maintenance costs \cite{nvidia_tco}.

To enable comparison with API models,the yearly \ac{tco} must be normalized to a cost-per-volume metric (e.g., Cost Per 1000 Prompts). This calculation is contingent on the hardware's achieved performance, measured in Requests Per Second (RPS) or Tokens Per Hour ($\text{TPH}$) \cite{pan2025_cost_benefit}. In our experiments, the cost per request is converted to cost per task by accounting for the number of model requests required to complete a full discussion.

\[
\text{Cost per Request} = \frac{\text{TCO}_{\text{Yearly Server Cost}}}{\text{Total Yearly Requests per Server}} \times 1000
\]
\noindent The $\text{Total Yearly Requests per Server}$ is a function of the Optimal RPS adjusted by the system's expected uptime fraction, which reflects the financial viability being directly tied to utilization rates \cite{nvidia_tco}. Although \ac{txo} is traditionally defined over annual periods, we amortize capital costs only over the actual experimental utilization window, resulting in a conservative estimate for short-term research deployments.


\paragraph{Proprietary models}
The task of calculating \ac{cpt} for proprietary models is significantly easier, since their service has a clear pricing structure. The \ac{cpt} for Proprietary LLMs ($CPT_{\text{Proprietary}}$) is calculated by separating the expense for the Input Sequence Length (ISL) and the Output Sequence Length (OSL), using the respective prices ($P_{\text{in}}$ and $P_{\text{out}}$) per million tokens.

\[
\text{CPT}_{\text{Proprietary}} = \frac{(\text{ISL} \times P_{\text{in}} + \text{OSL} \times P_{\text{out}})}{1,000,000} + C_{\text{API Overhead}} 
\]
\noindent where $C_{\text{API Overhead}}$ includes amortized costs from failed calls and vendor usage fees. For multi-turn discussions, ISL and OSL denote the cumulative input and output token counts across all model calls required to complete the task.


\paragraph{Human participation}

Human participation costs can not, and should not, be calculated in a vacuum. There are certainly ways to get participates with wages much lower than minimum wage for most countries, and there are even platforms such as Mechanical Turk which streamline this process. Besides the obvious and severe ethical implications of doing so, this often results in poor quality data \cite{Moss2023}. The true cost of procuring human labor in a research context is calculated using the (ethical) gross hourly wage, mandatory platform fees, and quality assurance overhead:

\[
\text{Cost Per Hour}_{\text{Human}} = W_{\text{gross}} \times (1 + F_{\text{platform}}) + C_{\text{QA-overhead}}
\]
\noindent where:
\begin{itemize}
	\item $W_{\text{gross}}$ is the ethical gross hourly wage (e.g., minimum recommended rate of \$8.00–\$12.00/hour).
	\item $F_{\text{platform}}$ is the commission charged by crowdsourcing platforms (e.g., $33.3\%$ for academic use on Prolific or $20-40\%$ on MTurk).
	\item $C_{\text{QA-overhead}}$ is the amortized internal labor cost incurred by the research team for quality control, screening, and dropout management.
\end{itemize}

The human labor rate is converted to $CPT_{\text{Human}}$ by estimating the time required to complete the specific research task:
\[
CPT_{\text{Human}} =(\text{Time Per Task}_{\text{est}} \times W_{\text{gross}}) \times(1 + F_{\text{platform}}) + C_{\text{QA per Task}}
\]


\subsection{The case for open-source models}

In order to minimize costs, we use open-source models hosted on on-premise servers. This setup allows us to drop the total costs of our experiments by two orders of magnitude compared with using a cheap human participants, and three times compared with closed source models (Table~\ref{tab:cost}). Additionally, using smaller, quantized LLMs~(\S\ref{ssec:experimental:setup}) allows us to run experiments on a single, consumer-grade GPU, nullifying the usual restriction of self-hosted LLMs.Note that we use more resources for the experiments in this paper, as our approach is partly exploratory, e.g., using different models for facilitators and users. Consequently, the open-source costs in Table~\ref{tab:cost} can be easily halved in practice.

Details around the methodology and assumptions of this result can be found in Appendix~\ref{sec:appendix:cost}.
